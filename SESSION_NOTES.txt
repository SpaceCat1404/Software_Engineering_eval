SESSION 2 (2026-07-14) — completeness checker rewritten to be numbering-free
-------------------------------------------------------------------------
Prompted by: Jeevan's submission genuinely has "6.1 Security Objectives"
and "6.2 Security Requirements" (own top-level Security section, not
nested under NFRs like the template's "5.1.x"), but was flagged MISSING
because the old checker only looked for section numbers "5.1.1"/"5.1.2"
exactly. With only 3 real submissions on hand, numbering isn't a reliable
enough signal to key off at all -- any section could plausibly be
renumbered by a student using a different template/tool.

srs_completeness.py (and the synced api/_lib copy) was rewritten to match
required sections by HEADING TEXT/KEYWORDS instead of numeric prefixes:
  - classify_heading() regexes each heading against 10 categories
    (introduction, overall_description, external_interfaces,
    functional_requirements, non_functional_requirements,
    quality_attributes, system_models, traceability_matrix, plus
    security_objectives/security_requirements handled separately since
    they can sit anywhere in the doc, not just nested under NFRs)
  - build_groups() replaces number-prefix matching: an anchor heading
    (matches a category) plus every following non-anchor heading up to
    the next anchor is that category's group, for word/row summing.
    Security objectives/requirements headings always close the current
    group so their content is never swept into the NFR count.
  - Found and fixed along the way: PDF exports sometimes use unicode
    dash variants (e.g. U+2011 in "Non‑Functional") that don't match a
    literal "-" in regexes -- added normalize() to collapse those before
    matching. This was silently making Jeevan's NFR count read 0 during
    the rewrite until caught by re-testing against all 3 real samples.

Re-ran against all 3 real submissions as a regression check: Jayanth's and
SRS_06's issue lists are byte-for-byte identical (just relabeled "where"
text), Jeevan's now correctly comes back 0 missing / 0 skimped (fr=21,
nfr=6, security_objectives=4, security_requirements=5) instead of 2
missing. batch_grade.py re-run to refresh pipeline_out/*.json (some of
those were stale from before an earlier NFR-counting fix and read nfr=0
where the checker now gives 6 -- not a live bug, just an unregenerated
file, but worth knowing if you see old numbers cached anywhere else).

Also: team detail page stat tiles used to show "21 / 15" which reads like
a fraction/percentage and is confusing at a glance. StatTile now takes an
optional `sublabel` prop; team detail page shows the found count as the
big value with "min 15" etc. as a small sublabel underneath instead of a
slash-ratio. Only that page used the ratio format -- dashboard and
/compare stat tiles already showed plain counts.

Two things noticed AFTER the above and worth flagging for continuity:

1. The web UI doesn't read pipeline_out/*.json at all -- it reads
   pipeline_out/blob_local/submissions/*.json (the local blob-store stand-in).
   That cache still had pre-rewrite results (Jeevan showing 2 missing) even
   after regenerating pipeline_out/ via batch_grade.py, because nothing had
   re-run /api/evaluate-local since the rewrite. Had to manually re-run the
   ingest+check+blob_store.put_json loop to refresh it. If completeness
   numbers ever look stale in the UI again, this cache -- not pipeline_out/
   -- is the one to check/regenerate (hit POST /api/evaluate-local, or run
   `python -m uvicorn api.index:app` and curl it).

2. Real ingestion bug in srs_ingest.py's PDF path, caught by manually
   reading SRS_06_Esection.pdf page-by-page after being asked to sanity-
   check a suspiciously-low 5/15 FR count (FR=5 turned out correct -- the
   student really only wrote 5 -- but NFR and Security Requirements were
   wrong). fitz's page.find_tables() runs per PAGE, so a requirements table
   whose rows spill across a page break comes back as two unrelated Table
   objects. The second page's first row is a DATA row, but pdf_table_to_dict()
   always treats a table's first row as its header -- so that data row got
   consumed as a header, corrupting the column mapping and silently
   dropping/mislabeling every row after it. Concretely: SRS_06's 5-row NFR
   table (PWM-NF-001..005) split 2+3 across pages 7/8 and was read as "2 NFR
   rows found"; its 4-row Security Requirements table (PWM-SR-001..004)
   split 1+3 and was read as "1". Fixed with _merge_as_continuation() in
   srs_ingest.py: if the most-recently-appended table in the current section
   has the same column count as a newly-found table, and that new table's
   first row doesn't contain a Req-ID-alias cell (i.e. doesn't look like a
   real header), its rows are appended to the previous table instead of
   parsed as a fresh one. Also had to skip rows with a blank Req-ID cell
   during that merge -- find_tables() sometimes emits a phantom row at the
   page boundary containing just the word-wrapped tail of the previous row's
   last cell (e.g. a lone "performance degradation." row), which isn't a
   real requirement.

   Verified against hand-counts from the raw PDF text for all three
   submissions after the fix -- SRS_06 now reads fr=5, nfr=5, security_
   objectives=4, security_requirements=4, all matching manual counts
   exactly (previously nfr=2, security_requirements=1). Jeevan's and
   Jayanth's counts were unaffected (Jeevan's FR/NFR/Security content is
   bullet-derived via regex, not PyMuPDF tables; Jayanth is a .docx, which
   never goes through the per-page PDF table path at all).

   Take the FR/NFR/security counts on ANY PDF submission you haven't
   personally read yet with a grain of salt until this fix has been through
   more real samples -- 3 submissions is not enough to be confident every
   page-break/table-layout edge case is covered.

SESSION 4 (2026-07-15, later) — real submission set (36 files), major
PDF heading-detection bug found and fixed, LLM-count trust softened
-------------------------------------------------------------------------
student_submissions/ restructured into subfolders by assignment type
(SRS_responses/, SAD_spec_responses/, test_plan_responses/) -- 36 real SRS
PDFs/docx now available in SRS_responses/, not just the original 3. User
pushed back (correctly) on trusting our own deterministic counts as
"ground truth" off a sample of 3 manually-checked files. Ran
batch_grade.py against all 36 as the real test of that concern, not more
manual spot-checks.

RESULT: 24 of 36 submissions came back with the *exact same* "10 missing,
2 skimped" signature despite being completely different projects --
obviously systemic, not per-document noise. Root cause: PDF_LEVEL1/2_MIN_SIZE
(srs_ingest.py's absolute-point-size heading heuristic, tuned only against
the original 3 docs) doesn't generalize. Checked several failing PDFs'
actual font metrics directly:
  - One doc's real headings sit at 13.0pt bold, PDF_LEVEL2_MIN_SIZE was
    13.5pt -- missed by half a point.
  - Two other docs have headings at 12.0pt -- IDENTICAL to their own body
    text size. No font-size signal exists in these documents at all. The
    only distinguishing feature is bold (PyMuPDF span flag bit 4) combined
    with the numbering pattern ("4.1 Authentication").
Any absolute (or even relative-to-body-size) font-size threshold was never
going to generalize across different Word->PDF export tools/settings --
confirmed by direct inspection, not assumption.

FIX in srs_ingest.py (+ synced api/_lib copy): added
_numbered_heading_level() (a short numbered line, <=10 words, "N" or
"N.M" or "N.M.K" prefix) as the PRIMARY heading detector, gated by
_is_bold() OR font size strictly greater than the document's own modal
body-text size (_document_body_size(), computed via a first pass over the
whole doc). The old PDF_LEVEL1/2_MIN_SIZE absolute-size checks are kept as
a secondary catch (still needed for title-page-style text with no
numbering) but no longer the primary/only signal.

Verified: regenerated all 3 original submissions -- Jeevan and SRS_06's
completeness counts are byte-identical to before this fix (0/0 missing-
skimped for Jeevan; same 5/5/4/4 counts for SRS_06) -- no regression.
Re-ran the full 36-submission batch: the "10 missing" clone pattern
dropped from 24 occurrences to 3. Manually checked all 3 remaining ones
and confirmed they are NOT ingestion bugs: "Lab1_SE...pdf" isn't an SRS
submission at all (it's a different lab's plain requirement table, wrong
file in the folder), and both "SRS - VITTHAL NAVALGUND...pdf" copies are
scanned/image-only PDFs with zero extractable text on every page checked
(needs OCR, which this pipeline doesn't have -- a real, separate,
not-yet-built capability, not a bug to "fix" today).

Also softened srs_score.py's prompt: previously told the model to treat
our parsed counts as "ground truth, do not recount" -- given we just found
a real bug in that same parser, that was overclaiming certainty. Changed
to "PARSED COUNTS ... generally reliable, but a heuristic, not infallible,
has been wrong before" and added an explicit instruction: if the model's
own reading of the text suggests a parsed count is off by a wide margin,
don't silently defer to it OR silently override it -- note a "COUNT
MISMATCH" in the flags field for a human to check, and use its own
estimate for scoring in that case. Tested against SRS_06 post-fix: it
did flag a mismatch (still thinks ~10 FRs vs the parser's 5) -- and in
this specific case the parser is actually the correct one (we hand-
verified 5 by reading the PDF page-by-page in session 3), so the model's
own count is the one that's wrong here. That's fine and expected -- the
point of this mechanism isn't to make the LLM's counting reliable, it's
to surface disagreement for a human to resolve via the "Document contents"
panel instead of either side silently winning.

Net effect for continuity: don't trust ANY count (deterministic or LLM) as
unquestionable on a submission nobody has manually read yet. The
deterministic parser is now meaningfully more robust after two real bugs
found and fixed this session (page-break table splitting, absolute-size
heading detection), but "meaningfully more robust" is not the same claim
as "correct." Keep expanding the batch-tested sample before trusting this
at grading scale.
